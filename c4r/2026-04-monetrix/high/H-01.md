# Incorrect decimals() implementation in sUSDM token
## Summary

The sUSDM token contract incorrectly defines its decimal precision as 6, whereas the protocol documentation specifies that it should be 12. This inconsistency can lead to incorrect calculations across the protocol and may impact critical financial logic.

## Vulnerability Details

According to the official documentation, the sUSDM token is expected to use 12 decimals:
https://doc.monetrix.xyz/guide/stake#how-the-exchange-rate-works

However, the contract explicitly overrides the decimals() function to return 6:

```solidity
File: sUSDM.sol
110:     function decimals() public pure override returns (uint8) {
111:         return 6;  //@audit-issue This is wrong.
112:     }
113: 
```

At the same time, the contract includes an internal method that suggests a decimal offset mechanism:
```solidity
File: sUSDM.sol
106:     function _decimalsOffset() internal pure override returns (uint8) {
107:         return 6;
108:     }
```

This shows clearly that the intended design is to rely on a base asset(USDM) decimal (6) combined with an offset (+6) to achieve an effective precision of 12. However, due to the explicit override of decimals(), this mechanism is effectively bypassed, and the token operates with only 6 decimals in practice.



## Impact
* Incorrect value representation:
Token amounts may be misinterpreted across the system due to mismatched decimal assumptions.
* Broken financial logic:
Any logic relying on 12-decimal precision (e.g., exchange rates, staking calculations, accounting) may produce incorrect results.
* Integration risks:
External integrations (frontends, oracles, or other contracts) that rely on documented behavior may malfunction.
* Potential loss of funds or accounting errors:
Miscalculations in balances, rewards, or conversions could lead to user-facing inconsistencies or financial discrepancies.

## Recommended Mitigation Steps
Align the contract implementation with the documented specification:
Either remove the decimals() override entirely, or update decimals() to return the intended value of 12.
