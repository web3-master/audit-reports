# Protocol Invariant Violation Due to Improper Handling of Removed Tradeable Assets
## Summary
The protocol allows the Governor to remove a tradeable asset without verifying whether it is still referenced in user vaults.
As a result, removed assets may continue to be included in solvency calculations, leading to inflated backing values and violation of core protocol invariants.

## Vulnerability Details
### Asset Removal Logic
The Governor can remove a tradeable asset via the following function:
```solidity
File: MonetrixConfig.sol
187: 
188:     function removeTradeableAsset(uint32 perpIndex) external onlyGovernor {
189:         require(isPerpWhitelisted[perpIndex], "Config: perp not listed");
190:         uint32 spotIdx = perpToSpot[perpIndex];
191:         uint32 pairAssetId = perpToSpotPairAssetId[perpIndex];
192: 
193:         // Swap-and-pop from the array
194:         uint256 len = tradeableAssets.length;
195:         for (uint256 i = 0; i < len; i++) {
196:             if (tradeableAssets[i].perpIndex == perpIndex) {
197:                 tradeableAssets[i] = tradeableAssets[len - 1];
198:                 tradeableAssets.pop();
199:                 break;
200:             }
201:         }
202: 
203:         delete perpToSpot[perpIndex];
204:         delete spotToPerp[spotIdx];
205:         delete perpToSpotPairAssetId[perpIndex];
206:         delete isPerpWhitelisted[perpIndex];
207:         delete isSpotWhitelisted[spotIdx];
208:         delete isSpotPairAssetIdWhitelisted[pairAssetId];
209:         emit TradeableAssetsUpdated(tradeableAssets.length);

```
This function does **not** check whether any vaults still hold positions or PM supplied assets associated with the removed asset.

### Solvency / Backing Calculation
The vault’s total backing value calculation depends on _readL1Backing():
```solidity
File: MonetrixAccountant.sol
137:     /// @dev Sum perp + spot USDC + spot hedge tokens + supplied (registered) + HLP for a single L1 account.
138:     function _readL1Backing(address account, SuppliedAsset[] storage suppliedList)
139:         internal view returns (int256 total)
140:     {
141:         total = _readPerpAccountValueSigned(account);
142: 
143:         // L1 spot USDC (idle cash not yet deployed to perp/hedge)
144:         total += int256(_readSpotUsdcBalance(account));
145: 
146:         // Supplied (0x811) — iterate only registered slots; strict reads.
147:         uint256 slen = suppliedList.length;                        //@audit-issue Any non-tradeable assets can be included in calculation, which is wrong!
148:         for (uint256 i = 0; i < slen; i++) {
149:             SuppliedAsset storage a = suppliedList[i];
150:             if (a.spotToken == uint64(HyperCoreConstants.USDC_TOKEN_INDEX)) {
151:                 total += int256(PrecompileReader.suppliedUsdcEvm(account));
152:             } else {
153:                 total += int256(
154:                     PrecompileReader.suppliedNotionalUsdcFromPerp(uint32(a.spotToken), a.perpIndex, account)
155:                 );
156:             }
157:         }
158: 
159:         // Spot hedge tokens (0x801) — unchanged; 0x801 returns 0 for unheld tokens rather than reverting.
160:         if (config != address(0)) {
161:             IMonetrixConfigReader cfg = IMonetrixConfigReader(config);
162:             uint256 len = cfg.tradeableAssetsLength();             //@audit-note Only tradeable assets are included in calculation.
163:             for (uint256 i = 0; i < len; i++) {
164:                 (uint32 perpIndex, uint32 spotIndex, ) = cfg.tradeableAssets(i);
165:                 total += int256(_readSpotAssetUsdc(spotIndex, perpIndex, account));
166:             }
167:         }
168: 
169:         // HLP equity counted at full mark value (no principal cap).
170:         // multisigVault-held HLP is recognized on the same basis as Vault-held HLP.
171:         total += int256(_readHlpEquity(account));
172:     }
173: 
```
**Issue**
* **Spot hedge assets** are included **only if they remain tradeable**, as they are iterated via tradeableAssets.
* **PM supplied assets**, however, are iterated from suppliedList **without validating whether they are still tradeable**.

This inconsistency leads to the following condition:
`If a tradeable asset is removed while still present in a vault’s suppliedList, its value will continue to be counted in the total backing calculation.
`

## Root Cause

The protocol lacks synchronization between:

* Asset lifecycle management (addition/removal of tradeable assets), and
* Accounting logic (which assumes all entries in suppliedList are valid and tradeable)

## Impact
* Protocol Invariant Violation:
The total backing value may be overstated, breaking the fundamental invariant that backing reflects only valid, tradeable assets.
* Incorrect Solvency Assessment:
Vaults may appear solvent when they are not, undermining liquidation and risk management mechanisms.
* Systemic Risk:
Inaccurate accounting at the vault level can propagate to protocol-wide inconsistencies, potentially leading to under-collateralization.

## Recommended Mitigation Steps
* Filter Non-Tradeable Assets:
Ensure that supplied assets are included in backing calculations **only if they remain tradeable**:
* Add Safety Checks on Removal:
Prevent removal of a tradeable asset if:
It is still referenced in any vault (positions or supplied assets).
