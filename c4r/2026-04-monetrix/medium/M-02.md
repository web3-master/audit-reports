## Title
Hedged Positions May Become Unclosable After Removal of Tradeable Assets

## Summary
The protocol allows the Governor to remove tradeable assets without verifying whether they are still referenced in existing hedged positions.
Since hedge management functions require assets to remain whitelisted, any positions tied to removed assets may become unclosable or unmodifiable, effectively locking them within the vault.

## Vulnerability Details
### Asset Removal Logic
The Governor can remove a tradeable asset using the following function:
```solidity
File: MonetrixConfig.sol
187: 
188:     function removeTradeableAsset(uint32 perpIndex) external onlyGovernor {
189:         require(isPerpWhitelisted[perpIndex], "Config: perp not listed");
190:         uint32 spotIdx = perpToSpot[perpIndex];
191:         uint32 pairAssetId = perpToSpotPairAssetId[perpIndex];
192: 
193:         // Swap-and-pop from the array
194:         uint256 len = tradeableAssets.length;
195:         for (uint256 i = 0; i < len; i++) {
196:             if (tradeableAssets[i].perpIndex == perpIndex) {
197:                 tradeableAssets[i] = tradeableAssets[len - 1];
198:                 tradeableAssets.pop();
199:                 break;
200:             }
201:         }
202: 
203:         delete perpToSpot[perpIndex];
204:         delete spotToPerp[spotIdx];
205:         delete perpToSpotPairAssetId[perpIndex];
206:         delete isPerpWhitelisted[perpIndex];
207:         delete isSpotWhitelisted[spotIdx];
208:         delete isSpotPairAssetIdWhitelisted[pairAssetId];
209:         emit TradeableAssetsUpdated(tradeableAssets.length);
210:     }
211: 
```
This function removes all whitelist mappings associated with the asset but **does not verify** **whether any vault still maintains active hedged positions** referencing it.

Both `closeHedge()` and `repairHedge()` rely on validation checks:
```solidity
File: MonetrixVault.sol
543:     /// @dev `spotAsset` is the HL limit-order asset for the spot leg (= 10000 + pair_index),
544:     ///      NOT the token_index. See `MonetrixConfig.TradeableAsset` for the distinction.
545:     function _requireHedgePair(uint32 perpAsset, uint32 spotAsset) internal view {
546:         require(config.isPerpWhitelisted(perpAsset), "perp not whitelisted");
547:         require(config.perpToSpotPairAssetId(perpAsset) == spotAsset, "spot/perp mismatch");
548:     }
549: 
550:     /// @dev For `isPerp=false`, `asset` is the HL pair_asset_id (= 10000 + pair_index).
551:     function _requireRepairAsset(uint32 asset, bool isPerp) internal view {
552:         if (isPerp) {
553:             require(config.isPerpWhitelisted(asset), "perp not whitelisted");
554:         } else {
555:             require(config.isSpotPairAssetIdWhitelisted(asset), "spot pair not wl");
556:         }
557:     }
```

**Issue**

Once a tradeable asset is removed:

* isPerpWhitelisted and related mappings are deleted.
* Validation checks in _requireHedgePair and _requireRepairAsset will fail.

As a result:

* Existing hedged positions referencing the removed asset **cannot be closed or repaired**.
* The protocol provides **no alternative mechanism** to manage these positions once the asset is delisted.

## Impact
* Position Locking:
Hedged positions may become permanently or indefinitely unmanageable (cannot be closed or repaired).
* Operational Risk:
Resolution depends on the Governor re-listing the asset, which may not occur in a timely manner or at all.
* Risk Exposure:
Inability to close or rebalance hedges can expose the vault to unintended market risk.

## Recommended Mitigation Steps
* Allow Management of Non-Whitelisted Assets:
Introduce mechanisms to close or repair positions even if the associated asset has been removed from the tradeable list.
